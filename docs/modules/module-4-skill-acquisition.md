- **Saved race reference (`CmRefV2`, 2026-06-20)** ‚Äî `CmPlan.cmRef` is a discriminated union. A **`kind:'cm'`** ref stores only the CM identity + **geometry** (`cmId`, `cmNumber`, `courseId`, `surface`, `distance`) and **derives** its conditions (ground/weather/season) from the **timeline** at read time, so a saved CM plan can never drift from the official conditions. A **`kind:'custom'`** ref stores geometry **and** its own conditions. `courseId`/`surface`/`distance` are common to both (readable without narrowing). Editing any condition of a CM race away from the timeline value flips the ref to `custom`. Legacy/foreign JSON is normalized on import by `normalizeCmRef` (cm refs keep geometry, drop stored conditions); Dexie migrated **v3 ‚Üí v4**. Full detail: [Race setup ‚Üê timeline SSOT](#race-setup--timeline-ssot-2026-06-20).

### Race setup ‚Üê timeline SSOT (2026-06-20)

The planner's race chooser now flows from the M3 **timeline** instead of a hardcoded preset list ‚Äî the timeline is the single source of truth for what each CM's track and conditions are. Spec: [2026-06-20-cm-race-timeline-reference-design.md](../superpowers/specs/2026-06-20-cm-race-timeline-reference-design.md) (+ amendment B); plan: [2026-06-20-cm-race-timeline-reference.md](../superpowers/plans/2026-06-20-cm-race-timeline-reference.md). Committed `8ee1f2e`..`9e21fae`.

- **`CmRefV2` union (decision B)** ‚Äî `{ kind:'cm'; cmId; cmNumber; courseId; surface; distance }` stores geometry and **derives** conditions from the timeline; `{ kind:'custom'; courseId; surface; distance; ground; weather; season }` stores everything. `courseId`/`surface`/`distance` are common to both, readable without narrowing. *Why store geometry on a `cm` ref rather than only the CM number?* The pure-cmNumber variant rippled into ~12‚Äì15 files (every `planToSimBuild` consumer would need timeline+catalog to resolve a course); storing geometry keeps the sim path synchronous and collapses the change to the UI layer ‚Äî only **conditions** derive. The earlier flat `cmRef` (courseId + all conditions stored) could silently drift from the official CM conditions; this union makes that impossible for `cm` refs.
- **Pure core** ([src/core/cmRace.ts](../../src/core/cmRace.ts)): `cmRaceOptions(entries)` (track-known CMs, recent-first), `normalizeCmRef` (classifies legacy/foreign refs ‚Äî cm keeps geometry, drops conditions), `currentCm(cmEntries, nowISO)`, `defaultConditions`. **Feature-layer mappers** ([cmRefSelection.ts](../../src/features/planner/race-setup/cmRefSelection.ts)): `cmRefToSelection` / `selectionToCmRef` (cmRef ‚áÑ `RaceSelection`) + `cmRefForEntry` (timeline entry ‚Üí `cm` ref). The mappers live in the feature layer because they touch `RaceSelection` + the course catalog; the pure data logic stays in `@/core`.
- **Single-state page** ‚Äî `CmPlannerPage` derives `selection` from `plan.cmRef` via `cmRefToSelection`; editing the race writes back through `selectionToCmRef({ ...plan, cmRef })`. Loading a saved plan re-derives the track automatically. The old dual-state `applyPlanTrackSetup` sync was deleted, removing the class of bugs where the displayed track and the saved `cmRef` disagreed.
- **`currentCm` everywhere** ‚Äî the timeline names the active CM: the first CM whose **end** date is ‚â• now (so a CM stays "current" through its whole run, not just up to its finals), falling back to the most recent past CM. `makeDefaultPlan(cmRefOverride?)` (first-run / legacy-migration / post-delete) and the **New** button both default a fresh plan to the **current** CM ‚Äî geometry resolved from the course catalog (loaded on demand; CM15 fallback before the timeline/catalog load) ‚Äî so the planner follows the live schedule instead of a hardcoded CM15.
- **Persistence** ‚Äî import (`parseCmPlan`) runs `normalizeCmRef`; Dexie schema bumped **v3 ‚Üí v4**.

## Purpose & boundary

M4 is the **pre-run planner for the ideal build** ‚Äî *"what should this build end up with, and is it reachable?"*. Discovery (what skills are worth targeting, ranked by real simulated length advantage) + sourcing (where to get them). **No SP constraint.**

It is **not** Module 2: M2 is the post-run, SP-limited min-max (pick the optimal affordable subset at a run's end). M4's L-ranking is the *discovery* engine; M2 does *purchase* optimization. See the [pipeline/boundary memory](../../CLAUDE.md) and spec ¬ß2.

The target skill list is an **output** of discovery, not an input ‚Äî that's the core redesign vs. the old coverage-matrix UI this supersedes (spec ¬ß1).

## Key mechanics & decisions

- **L = `evalSkillDelta(build, race, skillId, nsamples, seed?)`** ‚Äî runs the race with vs. without the skill on the same seed; `bashin = Œîposition / 2.5m`, averaged over Monte-Carlo samples. Returns `BashinStats {mean, median, min, max, nsamples, results}`.
- **`DISCOVERY_NSAMPLES = 30`** ‚Äî discovery trades precision for speed. Measured: 200 samples ‚âà 3.3 min for the full chart; 30 ‚âà 36 s with top rows streaming in ~1‚Äì2 s. (M2 sets its own higher count for precision.) TODO(1b): progressive refine of the surviving top-N.
- **`0 L` vs `n/a` (P3)** ‚Äî `nsamples === 0` ‚áí the engine **can't** evaluate it ‚Üí `n/a` (never a misleading `0 L`). `mean ‚â§ DEAD_L (0.1)` ‚áí `zero`. Both addable when "show every skill" is on.
- **Shared Kitasan default** ‚Äî first run, the New action, and the post-delete fallback use the same CM15 Kitasan baseline: `spd 1200 / sta 900 / pow 1000 / gut 600 / wit 1100`, Great mood, Turf A / Medium S / Front A. The untouched legacy empty/Pace starter is migrated automatically when it is the device's only saved plan. The engine **throws `firstPositionInLateRace` on a 0-speed runner**; the chart also guards `spd > 0` and shows an "enter your stats" prompt otherwise.
- **Aptitudes** read from `sparkGoals.pink` (default `A`), not `statProfile`.
- **Course geometry is embedded in the engine bundle** ‚Äî `resolveCourse(courseId)` needs no external `course_data.json` for the sim.

## Gotchas (M4-specific)

- **0-speed runner throws** `firstPositionInLateRace` ‚Äî never feed `evalSkillDelta` an all-zero build. `makeDefaultPlan` seeds non-zero stats; the chart guards `spd > 0`.
- **Pre-fix plans show the "enter stats" prompt** ‚Äî any plan created before the default-stats seed has all-zero stats, so `/` shows *"Enter your runner's stats (Speed is required)"* instead of ranking. Fix: enter stats in the Runner panel, or create a new plan.
- **Old coverage UI removed (2026-06-15)** ‚Äî the pre-engine `SkillPlannerPage` + `src/features/coverage/` + `src/features/inventory/` clusters (+ the `iconRetrofit` integration test that only exercised them) were deleted once the engine chart replaced them. `PlanHeaderPanel`/`SkillPicker` stayed (shared by the new page). Recover the coverage-matrix/inventory pattern from git history if Slice 1b needs it.
- **`runComparison` returns no aggregate stats** ‚Äî `min/max/mean/median` are NOT fields on the engine result; derive them by sorting the `results` array. Also: full-comparison `runData` carries per-frame traces for both runners; skill-comparison `runData` carries only `sk` activation logs ‚Äî do not conflate when extracting trace data.
- **Open `SkillDetailDisclosure` with `traceContext` crashes jsdom** ‚Äî constructing `SimClient` in jsdom spawns a real Worker, which jsdom can't handle. Any test that opens such a disclosure must `vi.mock('./useSkillTrace')` to stub the hook (see `UmaChartPanel.test.tsx`, `PlannerSidebar.test.tsx`).
